## [BUG-01] dotenv の読み込みタイミング問題（S3バケット名が空になる）

**発覚日:** 2026-05-09  
**影響範囲:** EC2上でアプリを起動したとき、`S3バケット: 未設定` と表示され、ファイル一覧取得に失敗する

### 原因

TypeScript は `import` 文を全てファイルの先頭に `require()` として展開する。  
そのため `import dotenv from 'dotenv'; dotenv.config()` と書いても、  
`import filesRouter from './routes/files'` の実行（＝ `routes/files.ts` の評価）が先に走り、  
`const BUCKET = process.env.S3_BUCKET_NAME || ''` が `''` で評価されてしまう。

### 修正内容

`package.json` の `scripts` に `-r dotenv/config` フラグを追加し、Node.js 起動前に dotenv をロードする：

```json
"dev": "ts-node-dev -r dotenv/config --respawn --transpile-only src/index.ts",
"start": "node -r dotenv/config dist/index.js"
```

あわせて `src/index.ts` から `import dotenv` と `dotenv.config()` を削除した。

## [BUG-02] 日本語ファイル名のアップロード時に文字化けが発生する

**発覚日:** 2026-05-09  
**影響範囲:** 日本語を含むファイル名でアップロードすると、S3上のキー名が文字化けする

### 原因

multer は `req.file.originalname` をブラウザが送信するままの Latin-1 として読み取る。  
ブラウザは実際には UTF-8 でファイル名を送っているため、そのまま使うと文字化けが発生する。

### 修正内容

`src/routes/files.ts` のアップロード処理で、ファイル名を UTF-8 に変換してから使用する：

```typescript
const filename = Buffer.from(req.file.originalname, 'latin1').toString('utf8')
```

## [BUG-03] multer v1 の非推奨警告が表示される

**発覚日:** 2026-05-09  
**影響範囲:** `npm install` 時および起動時にセキュリティ警告が表示される

### 原因

`package.json` の multer バージョンが `^1.x` を参照しており、v1 は deprecated。

### 修正内容

`package.json` の依存を更新：

```json
"multer": "^2.0.0"
```

TypeScript の型定義（`@types/multer`）は引き続き互換性あり。

## [BUG-04] Windows で `npm run dev` が実行できない（コマンド認識エラー）

**発覚日:** 2026-05-09  
**影響範囲:** WSL で `npm install` した後、Windows PowerShell から `npm run dev` を実行するとエラーになる

### 原因

WSL（Linux）環境で `npm install` を実行すると、Linux 用バイナリのみが生成される。  
Windows PowerShell から `ts-node-dev` や `vite` を呼ぶ際に必要な `.cmd` ファイルが作成されない。

### 修正内容

- **npm install は必ず Windows PowerShell から実行する**というルールを手順書に明記した
- `phase01/package.json` の `dev` スクリプトを `cd backend && npm run dev` 形式に変更した（`--prefix` 形式より安定するため）

## [BUG-05] EC2上で IMDSv1 の curl が応答しない

**発覚日:** 2026-05-09  
**影響範囲:** IAMロールの確認のため `curl http://169.254.169.254/...` を実行しても何も返らない

### 原因

EC2 インスタンスがデフォルトで IMDSv2 のみを受け付ける設定になっている。  
IMDSv1 形式（トークンなし）の curl はタイムアウトまたは空応答になる。

### 対処方法

IAMロールの確認は IMDSv2 形式で行う：

```bash
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/info
```

`"Code" : "Success"` が返れば IAMロールが正常にアタッチされている。

## [変更-01] Ubuntu 24.04 で `apt install awscli` が使えない

**発覚日:** 2026-05-09  
**影響範囲:** トラブルシューティング時の S3 接続確認コマンド

### 経緯

Ubuntu 24.04 では `awscli` パッケージが apt リポジトリから削除されている。

### 対処方法

AWS CLI の代わりに Node.js で直接 SDK を呼んで S3 接続を確認する：

```bash
cd ~/samurai-repo/phase01/backend
node -e "
const { S3Client, ListObjectsV2Command } = require('@aws-sdk/client-s3');
const client = new S3Client({ region: 'ap-northeast-1' });
client.send(new ListObjectsV2Command({ Bucket: '[バケット名]' }))
  .then(r => console.log('接続成功'))
  .catch(e => console.error('エラー:', e.name, e.message));
"
```

EC2 のハンズオン手順書のトラブルシューティング欄にも追記済み。
